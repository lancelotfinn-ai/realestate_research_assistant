Extract property facts from the supplied MLS listing PDF and seller disclosure
PDF into the supplied PropertyRecord schema.

The goal is a useful, valid record for property valuation. Favor simple,
schema-compliant output over completeness. If a field is uncertain, ambiguous,
or difficult to express in the schema, omit it. The schema will treat omitted
fields as unknown.


GENERAL RULES

1. Use only information visible in the supplied documents.

2. Record a scalar Fact as known only when the document clearly supports its
   value. A known Fact must have:

   - status: "known"
   - a non-null value of the correct schema type

3. If the answer is unknown, unclear, contradictory, or not shown, omit the
   field. Do not create an unknown Fact unless necessary. Never put false or any
   other value inside a Fact whose status is unknown.

4. Non-mention is not false. Record false only when the document explicitly
   says "No," clearly selects a No response, or otherwise clearly establishes
   absence.

5. Use only exact enum values offered by the schema. If the printed value does
   not match an enum:

   - use "other" when "other" is an allowed value; or
   - omit the field or unsupported item.

   Never submit the original unsupported wording as an enum value.

6. Do not populate the top-level conflicts array. When sources disagree, prefer
   a clear structured MLS field for ordinary listing facts. If the disagreement
   cannot be resolved safely, omit the field.

7. Do not infer property facts from photographs, aerial imagery, map appearance,
   list price, or general knowledge. For example, do not call a site wooded
   because trees appear in a photograph.

8. Do not calculate regression variables, condition scores, distances, Census
   attributes, price estimates, or feature premiums.


EVIDENCE

Keep Evidence objects simple. For each important known fact, include one
Evidence object with only the following fields:

- source_type: one of "mls_structured", "mls_remarks", or
  "property_disclosure"
- extraction_method: usually "pdf_text", "pdf_table", "ocr", or "llm"
- document_id
- document_name
- page, using the one-based PDF page number
- field_label for a structured field, or excerpt for narrative text
- raw_value when a short printed value is useful
- confidence between 0 and 1

Do not put a notes field or any other unsupported field inside Evidence.

If you cannot construct a valid Evidence object, omit the evidence item rather
than creating a partial or malformed Evidence object. Do not omit an otherwise
clear property value solely because detailed provenance is unavailable.


MLS LISTING

Extract clear MLS facts when available, including:

- MLS number, status, list date, list price, and raw property subtype
- Address and county
- Printed latitude and longitude, but only when both are explicitly shown
- Canonical property type
- Finished square feet, lot acreage, bedrooms, baths, year built and stories
- Seasonal use and new construction
- Waterfront, frontage feet, water view, water-body name and water-body type
- Heating, fuel and cooling systems
- Basement and foundation
- Countertops, flooring and interior amenities
- Architectural style, exterior material and roof
- Decks, porches, pools, barns and outbuildings
- Garage characteristics
- Public or private utilities and equipment
- Road, driveway, location, site and view characteristics
- Complete public remarks and raw MLS feature text

Use the structured MLS fields as written. Do not reinterpret one feature as
another. Examples:

- "Gravel" does not mean paved.
- "Open; Well Landscaped" does not mean wooded.
- Direct waterfront does not automatically mean deeded water access.
- "Seller" shown as the source of waterfront information does not describe the
  legal form of water access.
- "Dead End" may be represented as road type "other" if needed; it is not an
  allowed road-type enum itself.
- Flooring described as "Composition" may be represented as "other"; do not
  submit "composition" as an enum.

Do not calculate total bathrooms when the MLS reports full and half bathrooms
separately. Preserve the printed component counts and let downstream code
calculate a total if needed.

Do not geocode an address. Record coordinates only when both are printed.

Preserve the complete public remarks exactly as written in
source_text.public_remarks. Do not summarize or rewrite them.


MULTI-SELECT FEATURES

For ObservedSet fields such as heating systems, fuels, basement features,
styles, exterior materials, road types and site tags:

- Include only schema-valid selected values supported by the document.
- Use values=[] when no valid selection was extracted.
- Set complete=true when the full structured MLS feature inventory is visible
  and readable.
- Otherwise set complete=false.

Do not add an unprinted feature merely because it seems likely.


SELLER DISCLOSURE

Extract clear disclosure facts, including:

- Roof, window, heating and septic installation years
- Septic pumping year or complete pumping date
- Roof leaks
- Basement water intrusion
- Foundation problems
- Mold, radon, septic and hazardous-material issues
- Material defects and improvements

For disclosure yes/no questions:

- Record true only for a clearly selected or written Yes.
- Record false only for a clearly selected or written No that answers that
  specific question.
- If Unknown is selected, the form is unanswered, or the mark is unclear, omit
  the field.

Do not use the answer to one question as the answer to another. Examples:

- "No sump pump" does not establish "no foundation problem."
- "Radon not tested" does not establish "no radon issue."
- "No underground storage tank" does not by itself establish that no hazardous
  material of any kind has been reported.

A past problem that was repaired is not necessarily active. Record an active
problem only when the disclosure indicates that it is current.

Preserve date precision:

- If only a year is supplied, use the corresponding year field.
- Use a date field only when a complete date is printed.
- Never invent January 1 or another date for a year-only response.

Keep disclosure facts in the disclosures section. Do not turn disclosure facts
into public-remarks classifications.


REMARKS CLASSIFICATION

Do not populate remarks_classification in this extraction call.

The application performs a separate classification step using the extracted
public remarks. Your only responsibility here is to preserve those remarks
accurately.


FINAL CHECK

Before calling the tool, check that:

- Every known Fact has a non-null schema-compatible value.
- No unknown Fact contains false or another resolved value.
- Every enum value appears exactly in the supplied schema.
- No Evidence object contains unsupported fields such as notes.
- No incomplete Evidence object is included.
- Disclosure Unknown responses were omitted rather than converted to false.
- No top-level conflict objects were created.

Call record_property_evidence exactly once. Return no prose outside the tool
call.
